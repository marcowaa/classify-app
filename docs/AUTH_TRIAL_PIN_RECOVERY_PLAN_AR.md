# خطة معالجة عزل الوضع التجريبي واستبدال التوكن في مسارات المصادقة

تاريخ: 2026-03-22
الحالة: مكتمل

## 1) تعريف المشكلة
- بعض المستخدمين المسجلين يتم إبقاؤهم في سلوك جلسة تجريبية بسبب بقاء مفاتيح تصنيف trial في التخزين المحلي.
- شاشة PIN قد تتحول لمسار مغلق عند وجود familyCode حتى مع وجود جلسة حقيقية.
- زر الخروج في شاشة PIN لا يخرج المستخدم فعليا من نمط PIN (كان يمسح الجلسة ويُبقي familyCode).

## 2) الأهداف الوظيفية
- ضمان استبدال توكن الجلسة التجريبية بتوكن الحساب الحقيقي مباشرة بعد نجاح تسجيل الدخول.
- منع أي تصنيف trial قديم من التأثير على الجلسة الحقيقية.
- منع حبس المستخدم في شاشة PIN مع توفير خروج فعلي.
- الحفاظ على سلوك آمن ومتوقع لجميع المسارات (عادي + OTP + PIN + OAuth + ربط طفل تجريبي).

## 4) مصفوفة المسارات المنطقية الواجب تغطيتها

### أ) مسارات دخول الأب
- تسجيل جديد عادي (Email/Password):
  - متوقع: token أب + classification FULL + لا بقايا child trial.
- تسجيل دخول عادي مع OTP:
  - متوقع: بعد verify-otp يتم إنشاء جلسة أب كاملة، وأي child trial قديم لا يقيّد الكتابة.
- تسجيل دخول اجتماعي OAuth:
  - متوقع: تخزين token الأب، إزالة child trial، وضبط FULL.
- دخول الأب عبر PIN:
  - متوقع: إزالة أي child trial classification، وتأكيد جلسة الأب الحقيقية.

### ب) مسارات الطفل
- دخول طفل عبر PIN:
  - متوقع: تنظيف جلسة الأب القديمة قبل تفعيل childToken.
- بدء جلسة طفل تجريبية من age gate:
  - متوقع: childToken + CHILD_TRIAL فقط حتى يتم ربطه أو ترقية الحساب.

### ج) مسارات ربط الطفل التجريبي
- ربط تلقائي بعد دخول الأب:
  - متوقع: نجاح الربط => إزالة trialChildToken وبيانات الربط المؤقتة.
- فشل الربط:
  - متوقع: إشعار واضح + عدم كسر الدخول + إبقاء مسار المتابعة ممكنا.

### د) مسارات شاشة PIN
- وجود familyCode بدون أي token:
  - متوقع: عرض شاشة PIN.
- وجود familyCode مع token أب:
  - متوقع: الانتقال للوحة الأب، لا حبس في PIN.
- الضغط على زر الخروج من PIN:
  - متوقع: حذف familyCode وجميع الجلسات الفعالة والعودة لمسار الدخول.

## 5) أخطاء المستخدم المتوقعة وكيف يتم احتواؤها
- المستخدم يدخل PIN خاطئ عدة مرات:
  - سلوك متوقع: رسالة خطأ، إعادة تهيئة المدخلات، عدم تعليق الواجهة.
- المستخدم يبدّل بين حساب طفل وأب على نفس الجهاز:
  - سلوك متوقع: كل دخول ينظف مفاتيح الطرف الآخر لمنع تلوث الجلسة.
- المستخدم لديه بقايا localStorage من نسخة قديمة:
  - سلوك متوقع: مسارات الدخول الناجح تنظف المفاتيح المتعارضة تلقائيا.
- المستخدم يرجع للرابط الرئيسي بعد OAuth:
  - سلوك متوقع: الجلسة الحقيقية تبقى FULL ولا تعود لقيود trial.

## 6) معايير القبول
- لا يتم حظر عمليات الكتابة للحسابات الحقيقية بسبب trial flags قديمة.
- زر الخروج من شاشة PIN ينفذ خروج فعلي من النمط ولا يترك المستخدم في حلقة.
- أي تسجيل دخول حقيقي للأب يستبدل ويطهر أي child trial state.
- نجاح الفحوص: TypeScript + Build + Tests + Health.
